# Getting started — using the API

This page takes you from "I have an API key" to "I called the TrakRF API and got a `200` back" in about 10 minutes, using only standard HTTP tools. It mirrors the [UI quickstart](./ui) — pick whichever track matches your integration plan.

## What you'll need

- A TrakRF account and an API key. <EnvSignInLink>Sign in (or sign up) at the <EnvLabel /> app</EnvSignInLink>, then mint a key from the **avatar menu → API Keys → New Key** ([detail](../api/authentication#mint-your-first-api-key)). The token is shown once at creation — copy it immediately.
- An API client of your choice — `curl`, Postman, HTTPie, or your language's standard HTTP library. This guide uses `curl` in examples.
- 10 minutes.

## 2. Make your first call

Save your API key to an environment variable for the rest of this page:

```bash
export TRAKRF_API_KEY="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

The `/api/v1/locations/current` endpoint returns a snapshot of where TrakRF last saw each asset. It's cheap, requires `history:read`, and gives you a live signal that your key works. In the **New API Key** dialog, `history:read` corresponds to the **History** dropdown set to **Read** — see [Authentication → UI labels vs scope strings](../api/authentication#ui-labels) for the complete scope-string ↔ UI label mapping. If your key was minted without `history:read`, [mint a new one](../api/authentication#mint-your-first-api-key) — scopes can't be edited after creation.

```bash
curl -H "Authorization: Bearer $TRAKRF_API_KEY" \
     "$BASE_URL/api/v1/locations/current"
```

A successful response looks like:

```json
{
  "data": [
    {
      "asset_id": 4287,
      "asset_external_key": "ASSET-0001",
      "location_id": 42,
      "location_external_key": "LOC-0001",
      "last_seen": "2026-04-20T14:32:18Z"
    }
  ],
  "limit": 50,
  "offset": 0,
  "total_count": 1
}
```

The `data` array holds one item per asset that has ever been scanned. Each item carries both forms of the asset and location identifiers as flat scalar fields: integer `id` (server-assigned, canonical for path params and FK joins) and string `external_key` (your handle, useful for joining back to your system of record). See [Resource identifiers](../api/resource-identifiers).

If you get a `401`, the key is malformed or not being sent in the header. If you get a `403`, the body names the missing scope — for this endpoint it'll be `"Missing required scope: history:read"`. If you get a `429`, you're being rate-limited — see [Rate limits](../api/rate-limits). If you're calling from browser JavaScript and getting a CORS error with no body, the API is server-to-server only ([details](../api/authentication#server-to-server)).

All error bodies are wrapped in an `error` key — a `403` looks like:

```json
{
  "error": {
    "type": "forbidden",
    "title": "Forbidden",
    "status": 403,
    "detail": "Missing required scope: history:read",
    "instance": "/api/v1/locations/current",
    "request_id": "01JXXXXXXXXXXXXXXXXXXXXXXX"
  }
}
```

Write handlers against `body.error.type` and `body.error.detail`, not top-level fields. Full catalog: [Errors](../api/errors).

## 3. Interpret the response

The two key concepts integrators trip on:

- **Two identifiers per resource** — every asset and location has an integer `id` (server-assigned, used in URL path params and FK fields) and a string `external_key` (your handle for joining TrakRF records back to a SKU, asset tag, or ERP code). Both are first-class — pick whichever fits the context. Full convention: [Resource identifiers](../api/resource-identifiers).
- **Response envelope** — every endpoint on the public surface wraps payloads in `{ "data": ..., ... }`, including `GET /api/v1/orgs/me`. List endpoints add pagination metadata (`limit`, `offset`, `total_count`) — see [Pagination, filtering, sorting](../api/pagination-filtering-sorting).
