## Task 3: F2 — add cached-state caveat after "No client-side scrubbing required"

**Files:**
- Modify: `docs/api/quickstart.mdx` — insert one paragraph after line 131

The "No client-side scrubbing required" claim is true for the linear `curl GET | mutate | curl PATCH @-` pipeline immediately above it (verified end-to-end in BB35 cycle 3). It is **not** true for cached-state usage: an integrator who caches a GET, does any other PATCH on the same resource (which advances server-side `updated_at`), and then tries to PATCH the original cached body, hits a 400 `validation_error` / `read_only` on `updated_at`. The accept-if-matches rule (instant equality post-TRA-721) sees the drift and rejects.

The behavior is correct optimistic-concurrency control. The fix is a one-paragraph caveat that names *only* the field that drifts (`updated_at`) — not `created_at` (immutable after creation) or `deleted_at` (only changes on delete) — and frames the rule as a feature, not a quirk.

- [ ] **Step 1: Identify the exact insertion point**

Run: `sed -n '129,133p' docs/api/quickstart.mdx`
Expected (line 131 ends with "No client-side scrubbing required."):
```
### Round-trip: `GET` → mutate → `PATCH` {#round-trip-patch}

The minimal-form `PATCH` above sent only the field being changed. When your integration's data model mirrors the read shape one-to-one — you have an in-memory asset object you've been mutating — you'll want to send the whole thing back instead. Every read-only field on the read shape (server-managed `id` / `created_at` / `updated_at` / `deleted_at`, the `tags` collection, and the natural-key reference fields `external_key`, `location_id`, `location_external_key`, plus `parent_external_key` on locations) follows an [accept-if-matches, reject-if-differs](./resource-identifiers#read-shape-vs-write-shape) rule on `PATCH`, so echoing them straight back from a `GET` response is safe. No client-side scrubbing required.

```

## Task 4: F4 — narrow `missing_org_context` entry to `/orgs/me`

**Files:**
- Modify: `docs/api/errors.md:82`

Platform session determined empirically that `422 missing_org_context` can fire only on `GET /api/v1/orgs/me`. Every other operation hits a different terminal — a request whose principal has lost its org context will resolve as `403 forbidden` or `404 not_found` (per the resource), not 422, because every other operation requires an active org context to even reach the business logic that could surface the missing-org condition.

That makes the existing catalog entry slightly misleading: it reads as a general condition that could fire on any operation. The fix tightens the entry to call out the scope explicitly, which also closes the silent contradiction between the errors catalog and the spec (where 422 is declared only on `getCurrentOrg`).
