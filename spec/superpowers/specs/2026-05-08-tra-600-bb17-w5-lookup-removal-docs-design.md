# TRA-600 — BB17 W5 — `/lookup` removal docs sweep

## Goal

Catch the trakrf-docs site up to platform PR #271, which dropped
`/api/v1/{assets,locations}/lookup` in favor of a list-filter idiom:
`GET /api/v1/{assets,locations}?external_key=X`. The spec already
refreshed cleanly; this doc covers the prose-side rewrite across the
concept guides plus the spec/Postman commit.

No platform changes — the service is already correct. The work is
bringing every prose surface that says "use `/lookup`" into line with
the list-filter shape the service now serves.

## Background

Platform PR #271 removed the dedicated lookup operations because:

- `/{resource}/lookup` collided with `/{resource}/{id}` for non-GET
  methods (the original W5 finding).
- The list-filter idiom is what big-API players actually do (Stripe
  `?email=`, Shopify `?handle=`).
- The 0-or-1-in-array overhead is small relative to the
  routing-precedence and discoverability wins.

The new client recipe is one extra line of `jq`:

```bash
# Old (gone)
curl "$BASE_URL/api/v1/assets/lookup?external_key=SKU-7421-A"
# → 200 { "data": {asset} }  or  404

# New
curl "$BASE_URL/api/v1/assets?external_key=SKU-7421-A"
# → 200 { "data": [asset]?, "limit", "offset", "total_count" }
ASSET_ID=$(... | jq -r '.data[0].id // empty')
[ -z "$ASSET_ID" ] && echo "no match"
```

Response-shape change: lookup returned `{"data": {object}}`; list returns
`{"data": [array], limit, offset, total_count}`. Miss is an empty array,
not a 404.

### File 2: `docs/api/resource-identifiers.md`

The largest prose surface. Three edits.

#### Replace the `Natural-key lookup uses /lookup?external_key=` section (lines 26–44)

Rename the heading to **`Natural-key lookup uses ?external_key=`** to
keep parallel structure with the preceding **`Path-param lookup uses
id`** heading. The body is rewritten end-to-end:

```markdown
## Natural-key lookup uses `?external_key=`

When you have the natural key but not the canonical `id`, filter the
list endpoint by `external_key`:

\`\`\`bash
curl -H "Authorization: Bearer $TRAKRF_API_KEY" \
     "$BASE_URL/api/v1/assets?external_key=SKU-7421-A"
\`\`\`

The list envelope returns 0 or 1 matches in `data` (the partial unique
index `(org_id, external_key) WHERE deleted_at IS NULL` caps live rows
at one per key). Empty array is the miss signal — there is no 404 for a
natural-key miss on this filter. Soft-deleted rows are not addressable
through this filter; if you need to inspect a deleted record, look it
up by `id`.

\`\`\`bash
ASSET_ID=$(curl -sH "Authorization: Bearer $TRAKRF_API_KEY" \
     "$BASE_URL/api/v1/assets?external_key=SKU-7421-A" \
     | jq -r '.data[0].id // empty')
[ -z "$ASSET_ID" ] && echo "no asset with that external_key"
\`\`\`

The same shape is available on locations:

\`\`\`bash
curl -H "Authorization: Bearer $TRAKRF_API_KEY" \
     "$BASE_URL/api/v1/locations?external_key=BACK-STORAGE-2"
\`\`\`

Use the `?external_key=` filter when an integrator pastes an
`external_key` directly (a barcode scan, a CSV row, an ERP record) and
you need to resolve it to a TrakRF resource. Cache the returned `id` if
you'll touch the resource again — subsequent path-param reads avoid the
filter round trip.

Repeat the parameter to fetch any-of:
`?external_key=SKU-A&external_key=SKU-B` returns up to one match per
key (still capped by the live-row uniqueness rule), which is more
efficient than N parallel single-key requests when you're resolving a
batch.
```

The `(any-of repeat)` paragraph is new — it's a behavioral capability
the dedicated `/lookup` couldn't offer (it 400'd on multi-value), and
worth surfacing as the second-largest reason a reader picks this
filter over a path-param read.

#### Edit line 201 (`is_active` is authoritative example)

Drop `/lookup` from the still-resolves list, since the `?external_key=`


### File 4: `docs/api/errors.md`

#### Edit line 65 (`not_found` row in error catalog)

The list-filter miss is a 200 with empty `data[]`, not a 404. Update
the "When you'll see it" cell to drop the `/lookup` reference and
clarify what *does* trigger 404:

```markdown
| `not_found` | 404 | Resource lookup failed — path-param `id` doesn't resolve (`GET /api/v1/assets/99999`) or a sub-resource path doesn't exist (`GET /api/v1/assets/99999/history`). A list filter that resolves to zero rows returns 200 with empty `data[]`, not 404. | No — check the identifier |
```

The original cell conflated the `/lookup` 404 with the list-filter
empty result; both are gone now (lookup endpoint deleted; list filter
returns empty array). 404 is purely the path-param miss case.

#### Edit line 215 (idempotency retry recipe)

Replace `/lookup` recovery with the list-filter recovery:

```markdown
- **`POST /assets`, `POST /locations`** — retrying with the same
  `external_key` hits the partial unique index
  `(org_id, external_key) WHERE deleted_at IS NULL` and returns
  `409 conflict`. Detect the 409, then
  `GET /api/v1/{resource}?external_key=...` and read `.data[0].id` to
  recover the canonical `id`, then `PUT` to reconcile. **If you omit
  `external_key` on a POST `/assets` retry, you may create duplicates**
  — the server will mint a fresh `ASSET-NNNN` each time. For
  retry-critical workflows, always supply an `external_key`.
```
