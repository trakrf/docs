# TRA-580 — BB15 S-2/S-3 + C-1/C-2/C-3 docs follow-up

## Goal

Bring `trakrf-docs` into alignment with platform PR [#264](https://github.com/trakrf/platform/pull/264), which closed five BB15 findings on OpenAPI spec naming and enum semantics. Three are pure spec renames the refresh picks up (C-1, C-2, C-3), one is already-shipped enum metadata that the spec already carries (S-2), and one is doc-only prose (S-3).

## Background

The parent ticket groups five findings that share a single SDK regeneration cycle: rename `location.path` → `tree_path`, drop the `current_` prefix on the asset view's location FK pair, and rename the api-key create response's `data.key` → `data.token`. Platform shipped the service-side renames, regenerated the public spec, and verified that `error.type` already carries `x-extensible-enum: true` (S-2). One finding (S-3) was deferred to this docs session by user direction: prose-level acknowledgment that current codegen tools don't honor `x-extensible-enum`.

| Finding | Surface | Platform changed? | Docs needs update? |
| ------- | ------- | ----------------- | ------------------ |
| C-1 | `location.path` → `tree_path` | Yes (PR #264) | Yes — spec refresh + `resource-identifiers.md` (derived-field prose, JSON example, helper paragraph) |
| C-2 | api-key create `data.key` → `data.token` | Yes | No markdown surface — spec refresh only (programmatic mint isn't documented; see TRA-578) |
| C-3 | drop `current_` prefix on asset view FK pair | Yes | Yes — spec refresh + `resource-identifiers.md` (prose, JSON example, FK-pair table row, jq example, accept-on-write paragraph) |

| S-3 | codegen caveat for `x-extensible-enum` | N/A (doc-only) | Yes — paragraph in `versioning.md`, one-liner cross-reference in `errors.md` |

## Changes

### File 1: `docs/api/resource-identifiers.md`

#### C-3: drop `current_` prefix on asset FK pair

The asset view is "current" by being on the asset; the field name doesn't need to repeat that. After the rename, the asset response and the `GET /locations/current` row shape carry the same field names — an integration joining data from both endpoints no longer has to remap.

Edits:

- Line 58 prose: `current_location_id` (int) and `current_location_external_key` (string) → `location_id` (int) and `location_external_key` (string).
- Lines 66-67 JSON example: rename both keys.
- Line 82 FK-pair table row ("Present as `null`"): `current_location_id`, `current_location_external_key` → `location_id`, `location_external_key`.
- Line 89 read-vs-write prose: example field `current_location_external_key` → `location_external_key`.
- Line 122 jq example: `.current_location_external_key = "PORTABLE-1437"` → `.location_external_key = "PORTABLE-1437"`.
- Line 132 accept-on-write paragraph: `current_location_id` and `current_location_external_key` → `location_id` and `location_external_key`.

#### C-1: `location.path` → `tree_path`

The field rename removes the three-way collision with HTTP request path (`error.instance`, OpenAPI `paths`) and SDK params.path (openapi-fetch path-param object). The underlying ltree column and trigger semantics are unchanged.

Edits:

- Line 89 read-vs-write prose: derived fields list `path`, `depth` on locations → `tree_path`, `depth`.
- Line 130 read-only set paragraph: derived `path` and `depth` → derived `tree_path` and `depth`.
- Line 146 JSON example: `"path": "warehouse_west.back_storage_2"` → `"tree_path": "warehouse_west.back_storage_2"`.
- Line 161 helper paragraph: rename `path` → `tree_path` everywhere it refers to the ltree-derived label helper. Keep references to "the path" as natural English where they read as sentences (e.g. "contributes the segment ... to its descendants' paths" stays — it's referring to descendant tree-paths conceptually, not to a field name).
- Lines 163, 165: rename remaining field-name references.

#### Out of touch on this page

The "Locations: `parent_id` and `parent_external_key`" section heading stays — it's about the FK pair, not the helper field. The HTTP-path / `error.instance` / OpenAPI-paths senses of `path` elsewhere in the docs are unaffected.

### File 2: `docs/api/versioning.md`

#### S-3: codegen caveat for `x-extensible-enum`

The current Open enums section enumerates the markers and tells readers how to handle unknown values. It does not call out that the OpenAPI extension `x-extensible-enum: true` is informational under current codegen tooling — `openapi-typescript@7.13` and `openapi-generator-cli` (Go/Java/Python targets) emit the enum as a closed union and reject unknown values at runtime.

Add a paragraph to the **"Open (extensible) enums in v1"** subsection (after the bulleted list, around line 58) covering:

- The annotation is a semantic-intent marker, not a generator hint that today's tools honor.
- Generated TypeScript / Go / Java / Python clients will reject unknown enum values at parse / unmarshal time.
- The recommended workaround is the pattern in [How to handle unknown values in client code](#how-to-handle-unknown-values-in-client-code) — fall through on unknown values rather than relying on generator behavior.
- Frame as a known DX gap rather than a v1 contract change.

Tone follows the existing "Clients written against v1 will continue to work as TrakRF adds features. Clients that treat unknown response fields or unknown enum values as errors will not — so don't." voice.

### File 3: `docs/api/errors.md`

#### S-3 cross-reference

The Extensibility subsection (around line 81) tells readers `error.type` is extensible and to handle unknown values gracefully. Add one line after the existing paragraph: a brief note that the spec marker is informational under current codegen and a link to the new versioning paragraph for the substantive caveat.

Single sentence — the canonical explanation lives in versioning.md.

## Sequencing

Single PR, single spec-refresh cycle. Commit order following the TRA-579 convention:

1. design doc
2. plan doc
3. doc edits (C-1 + C-3 in `resource-identifiers.md`, S-3 in `versioning.md`, S-3 link in `errors.md`)
4. spec refresh as the last commit (`chore(spec): refresh openapi from platform@<sha>`)

That keeps spec-derived assertions in the doc commits checkable against the spec at the same SHA, mirroring how TRA-579 (#66) landed.
