# TRA-408 — API docs fixes (design)

**Linear:** [TRA-408](https://linear.app/trakrf/issue/TRA-408) — sub-issue of TRA-210
**Parent context:** Follow-ons from the 2026-04-20 black-box API evaluation. A new integrator opening the docs portal is currently sent down multiple dead ends before they can make their first successful API call.
**Related (not handled here):** TRA-407 (service-side contract fixes), TRA-409 (auto-publish spec on PR branches), TRA-395 (rate-limit behavior), TRA-393 (key-mint UI — *Done*), TRA-396 (read-path API-key auth — *Done*).

## Goal

A brand-new developer with a TrakRF login should be able to open the docs portal and get from "I have an account" to "I got a `200` back from the API" using only what the docs say — no reverse-engineering, no emailing support, no guessing endpoint names or URL shapes.

### §7 — `docs/api/resource-identifiers.md`: convention doc

- Intro: every resource has two IDs — the business `identifier` (string, human-meaningful, used in URLs) and the `surrogate_id` (integer, returned in response bodies, not required by clients).
- **URL path parameters** — reads take `{identifier}`. Example: `GET /api/v1/assets/ASSET-0001`. Integer `surrogate_id` is **not** accepted on reads (returns `404`). Side-by-side curl showing correct vs. incorrect.
- **Response bodies** — `surrogate_id` is present for joinability and debugging; clients should key on `identifier`.
- **Writes (PUT/DELETE)** — note that the spec currently shows `{id}` on write paths and will align with reads under TRA-407; this page will be updated when that lands. Clean hedge for the one TRA-407 dependency that surfaces here.
- **Session-auth-only exception** — `/api/v1/assets/by-id/{surrogate_id}/history` is an SPA-only form: API keys get `401`. Integrators should use `/assets/{identifier}/history`.
- Add to `sidebars.ts` `apiSidebar` after `rest-api-reference`.
- **Files:** 1 new, sidebar edit.
- **Expected diff:** ~70 lines.
