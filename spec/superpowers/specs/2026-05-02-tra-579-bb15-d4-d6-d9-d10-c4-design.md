After the field table (after line ~30, where `title` is described), add a brief sub-section that lists the canonical strings. This keeps the envelope-shape section authoritative without bloating the type catalog.

```markdown
### Canonical titles

`error.title` is fixed per `error.type`. Generated clients can rely on the pairing ŌĆö branch on `type`, log `title`, surface `detail` to humans.

| `error.type`             | `error.title`            |
| ------------------------ | ------------------------ |
| `validation_error`       | `Validation failed`      |
| `bad_request`            | `Bad request`            |
| `unauthorized`           | `Unauthorized`           |
| `forbidden`              | `Forbidden`              |
| `not_found`              | `Not found`              |
| `method_not_allowed`     | `Method not allowed`     |

| `unsupported_media_type` | `Unsupported media type` |
```

### File 3: `docs/api/resource-identifiers.md`

#### C-4: add `is_active` vs `valid_*` precedence

New short H2 section, placed after `## Asset external_key is optional` (around line 191) and before `## Tags use a composite natural key`. The placement keeps it adjacent to the assets-shape discussion where `is_active` and `valid_*` first appear in the response-shape table.

```markdown
## `is_active` is authoritative

Assets and locations carry both an `is_active` boolean and a pair of `valid_from` / `valid_to` timestamps. The API treats `is_active` as the source of truth for whether the resource is usable right now. `valid_from` and `valid_to` are informational metadata ŌĆö the v1 service does not compute effective state from them.

Concretely: an asset with `is_active: true` and `valid_to: 2024-01-01T00:00:00Z` (a `valid_to` already in the past) is still treated as active by the API. List filters on `is_active=true` will return it; `/lookup` will resolve it; PUTs will succeed.

If your business logic needs time-based filtering ("active and within its validity window"), apply that filter client-side. A computed `is_active_effective` field is on the v1.x roadmap if customer pain materializes; do not depend on the service deriving it today.
```

This is doc-only ŌĆö no service behavior changed. The point is to write down the v1 contract so integrators don't bake the wrong assumption into their client code.

## Out of scope

- D-4 docs change. The current resource-identifiers prose ("Multiple natural-key parameters or none returns `400`") is accurate post-fix. Adding the new specific detail string (`"exactly one of: external_key"`) would be helpful in principle but is the kind of low-leverage prose tweak the launch readiness sweep is explicitly avoiding. If a customer files a confusion ticket post-launch, revisit.
- Adding the canonical-title table to the type catalog. Keeping it in the envelope-shape section avoids a wide column on the catalog table and matches how `title` is conceptually grouped with the envelope, not the per-type retry behavior.
- Auditing the docs corpus for every literal `"Invalid request"`. The only ones that matter are the example envelopes that shape integrator expectations ŌĆö the four in `errors.md` and the one in `resource-identifiers.md`. CHANGELOG / commit-message uses don't affect contract.
- A computed `is_active_effective` field. Explicitly deferred per the C-4 decision.
- A `parent_id` worked example on the locations row of pagination. The pattern is already established by the assets `location_id` example; readers don't need a duplicate.
